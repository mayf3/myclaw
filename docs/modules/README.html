P0 全阶段 执行计划

这组文档是 MyClaw 的模块化 design review。每个模块都按同一套结构组织：参考项目观察、推荐设计、MVP 边界、依赖关系、风险和验收标准。新增 OpenHuman 和 OpenClaw 迁移目标后，初期方案从“先做完整 agent 平台”进一步收敛为“先做可观察的本地控制台 + 可恢复 workflow + 可审计迁移 plan”。

第一阶段必须完成

  • workflow-core.html 的 envelope、runner、resume 状态机。
  • tools-approval-security.html 的 workspace boundary、timeout、stdout cap、approval。
  • access-layer.html 的 CLI 命令。
  • config-state-storage.html 的 state 目录和 JSONL/index 存储。
  • openclaw-migration.html 的 dry-run plan 和 staged snapshot 设计。

其它模块先保留接口，不急着完整实现。

相关模块

参考项目对比比较 Hermes、OpenClaw、OpenHuman 与 Lobster，明确可借鉴设计和不应照搬的部分。 OpenHuman 参考分析拆解 OpenHuman 的 RPC、event bus、agent、记忆、渠道、skills、桌面壳和运维边界。 MyClaw 初期实现方案把三个参考项目收敛成 MyClaw 的 Node.js 初期实现切片、目录结构和验收顺序。 OpenClaw 一键迁移方案把 OpenClaw 一键迁移拆成 plan、stage、apply 三段，先做 dry-run inventory。 Workflow Core定义 workflow AST、runner、统一 envelope、approval/resume 状态机。 接入层拆分 CLI、Gateway、未来 channel adapter，确保入口层不承载业务逻辑。 Gateway本地 HTTP/WS 控制平面、token auth、run events、未来 UI 和渠道入口。 Agent RuntimeLLM provider、prompt builder、tool calling、transcript 和 approval pause。 工具、审批与安全ToolDescriptor、allow/deny、workspace boundary、dangerous action approval。 记忆、Session 与搜索JSONL、SQLite/FTS、memory files、session_search 和长期上下文召回。 插件与 Skills区分 runtime capability 与 prompt workflow，设计最小 plugin manifest。 配置、状态与存储JSON5 config、Zod strict schema、state 目录、secrets 与 SQLite 迁移。 控制台、观测与运维doctor/status、run timeline、approval queue、logs、未来 Control UI。
